### Inconsistencias Encontradas

**1. Nombre de negocio: dos versiones en uso**

`site.ts` define dos variantes del nombre del negocio:
- `brand.footerName` = "TC INSURANCE" (usado en el watermark del footer)
- `brand.name` = "TC Insurance Agency Services, LLC" (usado en copyright y legales)
- `business.name` = "TC Insurance Agency Services, LLC" (usado en schemas)

El watermark del footer es decorativo (`aria-hidden`) y no constituye un NAP real. Sin embargo, si un crawler o directorio extrae texto del DOM, podría registrar "TC INSURANCE" como nombre de la empresa. Recomendación: verificar que el GBP, Yelp y BBB usen siempre el nombre legal completo "TC Insurance Agency Services, LLC".

**2. Teléfono: campo `business.telephone` sin uso en ningún schema**

En `site.ts` línea 123, `business.telephone` = "+1 (203) 993-2369" (display), pero ningún componente de schema lo referencia. Todos los schemas usan `contact.phoneHref` = "+12039932369". El campo `business.telephone` es letra muerta y puede generar confusión en mantenimiento. Fix: eliminar `business.telephone` de `site.ts` o redirigirlo a `contact.phoneHref`.

**3. Teléfono no visible en el footer**

El footer no muestra el número de teléfono en ninguna sección visible. Para NAP completo, los crawlers de citaciones y Google esperan ver teléfono + dirección + nombre en el footer. Fix: añadir el número de teléfono al bloque "Oficina" del footer.

**4. Email: consistente pero informal**

El email `tcinsurance85@gmail.com` aparece en `site.ts` y en el schema `sobre-mi`. Es consistente, pero un email Gmail para una LLC puede reducir la autoridad percibida en directorios profesionales. Recomendación a mediano plazo: migrar a dominio propio (ej. `tatiana@tcinsurancetx.com`).

### Recomendados — Faltantes

**1. `openingHoursSpecification` (structured, reemplazar `openingHours` string)**

El campo `openingHours` como string plano es aceptado pero `openingHoursSpecification` como objeto structured es preferido por Google para rich results y GBP sync.

```json
"openingHoursSpecification": [
  {
    "@type": "OpeningHoursSpecification",
    "dayOfWeek": ["Monday","Tuesday","Wednesday","Thursday","Friday"],
    "opens": "09:00",
    "closes": "18:00"
  }
]
```

**2. `paymentAccepted`**

```json
"paymentAccepted": "Cash, Credit Card, Check, Zelle, Venmo"
```

**3. `hasOfferCatalog`**

Catálogo de servicios para rich results de servicios:

```json
"hasOfferCatalog": {
  "@type": "OfferCatalog",
  "name": "Servicios de Seguros",
  "itemListElement": [
    { "@type": "Offer", "itemOffered": { "@type": "Service", "name": "Seguro de Salud ACA" } },
    { "@type": "Offer", "itemOffered": { "@type": "Service", "name": "Seguro de Vida" } },
    { "@type": "Offer", "itemOffered": { "@type": "Service", "name": "Seguro de Gastos Finales" } },
    { "@type": "Offer", "itemOffered": { "@type": "Service", "name": "Medicare" } },
    { "@type": "Offer", "itemOffered": { "@type": "Service", "name": "Seguro Dental" } },
    { "@type": "Offer", "itemOffered": { "@type": "Service", "name": "Seguro de Visión" } },
    { "@type": "Offer", "itemOffered": { "@type": "Service", "name": "Seguro de Indemnización" } }
  ]
}
```

**4. `aggregateRating`**

Con 6 testimonials con rating (5 con 5 estrellas, 1 con 4), se puede construir un AggregateRating válido. Google muestra las estrellas en los resultados de búsqueda cuando este campo está presente.

```json
"aggregateRating": {
  "@type": "AggregateRating",
  "ratingValue": "4.8",
  "reviewCount": "6",
  "bestRating": "5",
  "worstRating": "1"
}
```

**5. `review` (reviews individuales con schema)**

Los testimonials en `src/data/testimonials.ts` tienen todos los campos necesarios (quote, name, rating, service) pero no se generan como schema `Review`. Añadir un array de reviews al JSON-LD aumenta la probabilidad de rich snippets de estrellas.

**6. `serviceArea` con `AdministrativeArea`**

La propiedad `areaServed` actual lista ciudades individuales pero no define un área de servicio geográfica explícita. Añadir:

```json
"serviceArea": {
  "@type": "AdministrativeArea",
  "name": "Dallas-Fort Worth Metroplex, Texas"
}
```

**7. `knowsLanguage` (nombre correcto del campo)**

El schema actual usa `availableLanguage` que es correcto para Organization. Sin embargo, para `InsuranceAgency` como subtipo de `LocalBusiness`, el campo `knowsLanguage` es más semánticamente preciso. Mantener `availableLanguage` pero añadir `knowsLanguage` también.

**8. `accreditation` / licencia de Texas**

```json
"hasCredential": {
  "@type": "EducationalOccupationalCredential",
  "credentialCategory": "Insurance License",
  "recognizedBy": {
    "@type": "Organization",
    "name": "Texas Department of Insurance"
  }
}
```

**9. LinkedIn en `sameAs`**

`siteConfig.business.sameAs` filtra nulos correctamente, pero LinkedIn no está configurado. Crear perfil de empresa en LinkedIn y añadir la URL como `NEXT_PUBLIC_LINKEDIN_URL` en el env.
