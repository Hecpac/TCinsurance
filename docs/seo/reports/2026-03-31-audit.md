### 2. Global `LocalBusinessJsonLd` WebPage node always points to `/#home` regardless of current page

- **Pages:** All pages except homepage (24 of 25)
- **Impact:** The `@graph` in `LocalBusinessJsonLd.tsx` includes a `WebPage` node hardcoded to `@id: "https://tcinsurancetx.com/#home"` and `url: "https://tcinsurancetx.com"`. When this schema is injected into `/servicios`, `/sobre-mi`, `/blog/[slug]`, or any other page, Google's Rich Results processor sees a `WebPage` entity claiming it belongs to the homepage URL. This means every page is asserting "I am the homepage" to the structured data graph, which can suppress page-level entity recognition and prevent accurate site graph construction. This affects all 24 non-homepage pages.
- **Current state:** `src/components/LocalBusinessJsonLd.tsx` line 73–80 defines the `WebPage` node with a static `homePageId` (`/#home`) and `url: siteConfig.seo.siteUrl`. This is injected globally via `src/app/layout.tsx`.
- **Suggested fix:** Remove the `WebPage` node from the global `LocalBusinessJsonLd` component. Keep only `InsuranceAgency` and `WebSite` nodes in the global schema. Each individual page (via `generateMetadata`) already handles its own canonical and page identity. Alternatively, make the WebPage node dynamic by passing the current URL as a prop — but removal is cleaner since Next.js metadata already signals page identity.

```tsx
// src/components/LocalBusinessJsonLd.tsx — remove WebPage from @graph
const jsonLd = {
  "@context": "https://schema.org",
  "@graph": [
    {
      "@type": siteConfig.business.type,
      "@id": organizationId,
      // ... all InsuranceAgency fields
    },
    {
      "@type": "WebSite",
      "@id": websiteId,
      name: siteConfig.brand.name,
      url: siteConfig.seo.siteUrl,
      inLanguage: "es-US",
      publisher: { "@id": organizationId },
    },
    // Remove the WebPage node entirely from global schema
  ],
};
```

## P1 — Important Issues

### 3. Homepage canonical URL missing trailing slash (inconsistency with `fullUrl`)

- **Page:** `/`
- **Impact:** The crawl records `fullUrl: "https://tcinsurancetx.com/"` (with trailing slash, as served) but `canonical: "https://tcinsurancetx.com"` (without trailing slash). While Google typically treats these as equivalent and has canonicalization heuristics, serving the canonical without a trailing slash while the page loads on the trailing-slash URL can generate soft duplicate signals. The sitemap also generates the URL without a trailing slash for `/`. Consistency is best practice.
- **Current state:** `src/app/layout.tsx` metadata has `alternates: { canonical: "/" }`. Next.js resolves this as `https://tcinsurancetx.com` (no trailing slash), but the served URL path is `/`.
- **Suggested fix:** This is expected Next.js behavior and likely fine. However, to make it explicit, verify that `https://tcinsurancetx.com` redirects to `https://tcinsurancetx.com/` (or vice versa) consistently in the hosting environment (Vercel) with a 301. No code change needed if the canonical form is consistently `https://tcinsurancetx.com` everywhere (sitemap + canonical tag should agree on the slash pattern).

### 4. Article `keywords` and `about.name` fields use raw category labels instead of actual keywords

- **Pages:** 7 of 9 blog posts (all except `seguro-gastos-finales-texas-guia-completa` which has proper keywords)
- **Impact:** The `keywords` field on the Article schema and the `about.name` on the `InsuranceProduct` node are set to the post category string (e.g., `"SALUD"`, `"VIDA"`, `"MEDICARE"`, `"NOTICIAS"`). These are internal taxonomy labels, not meaningful keyword phrases. Google does not penalize this but loses an opportunity to understand topical relevance, especially for `about.name` where a proper `InsuranceProduct` name would strengthen E-E-A-T signals.
- **Current state:** `src/app/blog/[slug]/page.tsx` lines 262–266 set `keywords` and the `about.name` using `post.category` for all posts except the hardcoded gastos-finales exception.
- **Suggested fix:** Add a `keywords` field to the blog post data structure in `src/data/blogPosts.ts`, or derive it from existing metadata. At minimum, the `about.name` should reference the actual insurance product topic:

```ts
// In blogPosts.ts data structure, add:
// keywords: "seguro de salud texas, planes HMO PPO Texas, asesoría seguros español"

// In blog/[slug]/page.tsx:
about: {
  "@type": "InsuranceProduct",
  name: post.aboutProduct ?? post.category,  // use a specific product name
},
keywords: post.keywords ?? post.category,
```

### 5. LCP not measured on any page (all values are `null`)

- **Pages:** All 25 pages
- **Impact:** Largest Contentful Paint is one of Google's three Core Web Vitals and directly affects ranking. The crawl script is not capturing LCP, meaning there is no automated monitoring for this critical metric. The homepage serves hero video (`hero-video-desktop.mp4` and `hero-video-mobile.mp4`) which is a known LCP risk — videos are typically not eligible as LCP candidates and can cause the real LCP element to be a large image or text block that loads late.
- **Current state:** All pages show `"lcp": null` in timing data. The homepage has the highest wall clock load time at 3,355ms (vs. ~1,100–1,750ms for other pages), likely due to video and hero image loading. The hero video poster image is `hero-portrait.jpg` but it is not confirmed as using `priority` prop.
- **Suggested fix (two parts):**
  1. Update the crawl script to capture LCP via `PerformanceObserver` or use the Navigation Timing API `largest-contentful-paint` entry type.
  2. Ensure the hero section's largest above-the-fold image uses `<Image priority />` (Next.js) so it gets `<link rel="preload">` in the response. Verify the poster image for the hero video is preloaded.

### 6. `dateModified` is always equal to `datePublished` on all blog posts (no update tracking)

- **Pages:** All 9 blog posts
- **Impact:** Google uses `dateModified` to determine freshness for ranking queries where recency matters (health/insurance news searches). Static `dateModified = datePublished` signals that content is never updated, which reduces freshness scores for time-sensitive insurance topics (ACA subsidies, Medicare changes, Medicaid renewals — all 2026 topics).
- **Current state:** `src/app/blog/[slug]/page.tsx` line 235: `dateModified: \`${post.publishedAt}T12:00:00-06:00\``. The `blogPosts.ts` data structure has only `publishedAt`, no `updatedAt` field.
- **Suggested fix:** Add an optional `updatedAt` field to the blog post data type and use it when present:

```ts
// In blogPosts.ts type definition:
updatedAt?: string;  // "YYYY-MM-DD" format

// In blog/[slug]/page.tsx:
dateModified: `${post.updatedAt ?? post.publishedAt}T12:00:00-06:00`,
```

- **Pages:** All pages (via `LocalBusinessJsonLd`)
- **Impact:** The schema uses `"openingHours": "Mo-Fr 09:00-18:00"` as a string. The string format used is valid for Google's Local Business rich result, but using a structured array ensures cross-platform compatibility.
- **Current state:** `src/components/LocalBusinessJsonLd.tsx` line 51: `openingHours: "Mo-Fr 09:00-18:00"`.
- **Suggested fix (minor):** Wrap in an array: `openingHours: ["Mo-Fr 09:00-18:00"]`. This matches Google's documented format for `openingHours` as a string array.
